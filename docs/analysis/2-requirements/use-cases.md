# Сценарии использования (Use Cases)

## Этап 2. Сценарии использования. Обязательно прорабатываем альтернативные потоки — ИИ любит их забывать.

**Скоуп:** клиентское приложение и API для него. Слоты, программы и шефы поступают из существующего бэкенда через API (read-only).

---

## UC-1: Запись на класс (себя и гостей)

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован; в расписании есть хотя бы один слот со свободными местами.
- **Триггер:** Клиент хочет забронировать место на кулинарный класс.

**Бизнес-решение по составу гостей (R-013).** По гостям собираем только количество мест и выбор экипировки (своя/прокатная). Имена, контакты, возраст и иные данные гостей в MVP не собираются; ответственность за гостей несёт клиент, оформивший бронь.

**Бизнес-решение по идемпотентности (R-022).** Клиент обязан передавать Idempotency-Key при создании брони. Повтор той же операции (тот же ключ и то же тело запроса) даёт идентичный ответ и не создаёт дубль брони. Повтор с тем же ключом, но другим телом — ошибка.

**Время и источник истины (R-021).** Время хранится и передаётся в UTC, отображается в локальной зоне студии. Сервер — источник истины по времени и статусам; при расхождении клиентского прогноза и серверного ответа применяется серверный статус.

### Основной поток

1. Клиент открывает список слотов.
2. (Опц.) Применяет фильтры: дата/период, тип программы (простая/сложная), наличие мест, шеф.
3. Открывает карточку слота и просматривает параметры (дата/время, программа и её тип, описание меню, шеф, цена, всего/свободно мест, доступность прокатной экипировки, адрес студии на карте).
4. Указывает число мест (себя + до 2 гостей). По гостям указывается только количество (R-013).
5. Для каждого места выбирает вариант экипировки: своя (фартук + ножи) / прокатная.
6. (Опц.) Указывает текстовый комментарий к брони — аллергии и особенности питания (FR-50).
7. Подтверждает запись. Клиент формирует и передаёт Idempotency-Key для этой операции (R-022).
8. Система проверяет лимит мест `max_seats = min(free_seats, program.capacity_cap, 3)` (три фактора: свободные места, потолок программы, лимит 3 на бронь) и отдельно прокатный фонд `rental_count ≤ free_rental_kits`, создаёт запись, уменьшает счётчики свободных мест/прокатных комплектов экипировки.
9. Система показывает подтверждение с ценой (оплата офлайн) и добавляет запись в «Мои бронирования».

### Альтернативные потоки

- **A1. Все экипировки — свои.** На шаге 5 прокатный фонд не затрагивается; уменьшается только число мест в группе.
- **A2. Часть мест — прокат.** На шаге 8 проверяется и уменьшается прокатный фонд только на число «прокатных» комплектов.
- **A3. Фильтр не дал результатов.** На шаге 2 список пуст — система предлагает сбросить/изменить фильтры.
- **A4. Клиент указал комментарий про аллергии.** На шаге 6 комментарий сохраняется вместе с бронью и передаётся в админку/шефу; на расчёт цены и лимиты не влияет.

### Исключения / ошибки

- **E1. Не хватает мест в группе.** Запрошено больше, чем свободно мест → запись отклоняется, показывается актуальное число свободных мест.
- **E2. Не хватает прокатной экипировки.** Прокатных мест запрошено больше, чем свободно комплектов → система предлагает уменьшить число прокатных мест или выбрать «своя экипировка».
- **E3. Гонка запросов / слот заполнился параллельно.** На шаге 8 места уже заняты другим клиентом → запись отклоняется без овербукинга и двойной брони (NFR-8), список обновляется.
- **E4. Сетевой сбой / неопределённый результат при подтверждении (R-020, R-022).** Запись не создаётся частично; при неопределённом результате клиент повторяет операцию с тем же Idempotency-Key, повтор не приводит к задвоению (NFR-9). Мутации (создание брони) офлайн запрещены — при отсутствии сети показывается понятное сообщение, кнопка подтверждения недоступна.
- **E5. Слот отменён или уже стартовал (R-023).** Слот к моменту подтверждения отменён студией или класс начался → запись отклоняется с соответствующим кодом (`slot_cancelled` / `slot_started`).

### Матрица ошибок ключевых операций (R-023)

Канон таксономии — коды API. Источник локализованного текста — клиентское приложение (по коду ошибки и полям `details`).

| Код HTTP | Обязательные details | UX-реакция | Retry |
| :--- | :--- | :--- | :--- |
| `slot_full` | 409 | `available_seats`, `available_rental_kits` | Показать актуальные свободные места/комплекты экипировки, обновить карточку | Нет (изменить число мест) |
| `double_booking` | 409 | `booking_id` | Сообщить, что бронь на слот уже есть; перейти в «Мои бронирования» | Нет |
| `slot_cancelled` | 410 | `slot_id` | Сообщить об отмене слота студией, убрать CTA записи | Нет |
| `slot_started` | 422 | `slot_id`, `start_at` | Сообщить, что класс начался/прошёл, запись недоступна | Нет |
| `invalid_code` | 400/401 | — | Сообщение «неверный код», поле OTP остаётся активным | Да (ввод заново) |
| 401 (unauthorized) | 401 | — | Очистка токена, переход на авторизацию (R-016) | Да (после входа) |
| 403 (forbidden) | 403 | — | Сообщение об отсутствии прав; действие недоступно | Нет |
| 429 (rate limit) | 429 | `retry_after` | Сообщение «слишком много попыток», отложенный повтор | Да (после `retry_after`) |
| 5xx (server error) | 5xx | — | Единый Error/Retry; для брони — повтор по тому же Idempotency-Key | Да |

### Постусловия

- **При успехе:** создана запись клиента; счётчики слота и прокатного фонда уменьшены; запись видна в «Мои бронирования»; комментарий про аллергии сохранён.
- **При ошибке:** состояние слота не изменилось.

## UC-2: Отмена записи

- **Актёры:** Клиент
- **Предусловия:** У клиента есть предстоящая запись; класс ещё не стартовал.
- **Триггер:** Клиент решает отменить участие.

**Бизнес-решение по объёму отмены (R-014).** Отмена выполняется только целиком: отменяется вся бронь со всеми местами и прокатными комплектами экипировки. Частичная отмена (например, одного гостя) не поддерживается; при ранней отмене освобождаются все места и прокатные комплекты брони.

**Время и источник истины (R-021).** Время хранится и передаётся в UTC, отображается в локальной зоне студии. Сервер — источник истины по времени и по статусу отмены.

**Граница политики:** ровно 24 часа до старта = ещё ранняя (бесплатная) отмена. При расхождении клиентского прогноза (ранняя/поздняя) и серверного ответа применяется серверный статус.

### Основной поток

1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Нажимает «Отменить» (отмена всей брони целиком, R-014).
3. Сервер определяет время до старта (источник истины — серверное время, R-021).
4. Ранняя отмена (≥ 24 ч до старта): вся бронь помечается «отменена»; все забронированные места и прокатные комплекты экипировки возвращаются в слот.
5. Система показывает подтверждение отмены.

### Альтернативные потоки

- **A1. Поздняя отмена (< 24 ч до старта).** На шаге 4 запись помечается «поздняя отмена»; места НЕ освобождаются; денежных штрафов нет; клиенту показывается пояснение политики 24 часов.
- **A2. Расхождение клиентского прогноза и сервера (R-021).** Клиент показал «ранняя отмена», а сервер вернул «поздняя» (или наоборот) → применяется и отображается серверный статус.

### Исключения / ошибки

- **E1. Класс уже стартовал/прошёл (`slot_started`).** Кнопка отмены недоступна; запись нельзя отменить.
- **E2. Запись уже отменена (`already_cancelled`).** Повторная отмена не выполняется; показывается текущий статус.
- **E3. Отмена офлайн / неопределённый результат (R-020).** Мутация отмены офлайн запрещена — при отсутствии сети показывается понятное сообщение, кнопка недоступна; при неопределённом результате применяется единый Error/Retry, финальный статус берётся с сервера. Коды ошибок и UX-реакции — по матрице ошибок UC-1 (R-023).

### Постусловия

- Статус всей брони обновлён («отменена» / «поздняя отмена»).
- При ранней отмене все места и прокатные комплекты экипировки брони снова доступны другим клиентам.

## UC-3: Просмотр и фильтрация списка слотов

- **Актёры:** Клиент
- **Предусловия:** Клиент авторизован.
- **Триггер:** Клиент хочет найти подходящий кулинарный класс.

### Основной поток

1. Клиент открывает список слотов. По умолчанию (`only_available=false`) показываются слоты на ближайшие 7 дней (дефолт API: `date_from=now`, `date_to=now+7 дней`; больший период — явным фильтром дат, R-027), отсортированные по времени старта (по возрастанию).
2. Заполненные (`free_seats=0`) и отменённые слоты отображаются в списке с пометкой «Мест нет» и неактивной (disabled) CTA записи (см. R-012).
3. (Опц.) Задаёт фильтры: дата/период, тип программы (простая/сложная), наличие мест, шеф.
4. Система показывает отфильтрованный список с ключевыми параметрами каждого слота, сохраняя сортировку по времени старта.

### Альтернативные потоки

- **A1. Сброс фильтров.** Клиент очищает фильтры → возвращается дефолтная выдача (слоты на ближайшие 7 дней, дефолт без фильтров).
- **A2. Только свободные слоты.** Клиент включает фильтр наличия мест (`only_available=true`) → заполненные/отменённые слоты скрываются.
- **A3. Семантика комбинирования фильтров (FR-38).** Внутри одной группы значения комбинируются по «ИЛИ» (`program_type`, `chef_id` — массивы), между разными группами — по «И». Границы периода дат включительные; ввод и трактовка дат — в локальной зоне студии. Активные фильтры отображаются клиенту.
- **A4. Пустой справочник шефов.** Если список шефов пуст, фильтр по шефу скрывается.

### Исключения / ошибки

- **E1. Нет слотов под условия.** Показывается пустое состояние с подсказкой изменить/сбросить фильтры.
- **E2. Нет расписания вовсе.** В горизонте выдачи нет ни одного предстоящего слота → показывается пустое состояние «Классов пока нет».
- **E3. Просмотр офлайн (R-020).** При отсутствии сети показывается ранее загруженный кэш списка с пометкой «данные могли устареть»; единый Error/Retry для повторной загрузки.

### Постусловия

- Клиент видит актуальный (или помеченный как устаревший) список предстоящих слотов; данные расписания не изменяются.
