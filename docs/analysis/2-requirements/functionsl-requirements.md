# Функциональные требования

## Этап 2. Что система должна делать (функции, поведение). Не путать с нефункциональными (non-functional-requirements.md).

Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

**О нумерации требований.** ID требований (FR-NN) сохраняются стабильными для трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре). Нумерация намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных документах. Новые требования продолжают сквозную схему FR-NN без переиспользования занятых номеров.

---

## Границы скоупа (клиентское приложение + API)

Скоуп — клиентское мобильное приложение и API для него. Функции инструктора, владельца и администратора обеспечиваются существующей инфраструктурой и здесь не описываются.

**Следствия для скоупа:**
- Слоты (классы) приходят из существующего бэкенда через API. Экранов создания и редактирования расписания в клиентском приложении нет.
- Программы (меню) и шефы доступны клиенту как справочные (read-only) данные из API — клиент видит их в карточке слота, но не управляет ими.

**Семантика фильтров (FR-38).** Внутри одной группы значения объединяются по ИЛИ (OR): `program_type` и `chef_id` — массивы значений (например, выбраны и простая, и сложная программа → показать оба). Между разными группами фильтров действует И (AND) (например, тип программы И период дат И шеф). Границы периода дат — включительные; даты вводятся в локальной дате клиента. По умолчанию фильтры не заданы (показывается дефолтная выдача — слоты на ближайшие 7 дней, см. FR-9 и R-027). Если справочник шефов пуст, фильтр по шефу скрывается. Активные (применённые) фильтры отображаются клиенту.

| **FR-14** | Система должна вести отдельный учёт прокатного фонда экипировки независимо от учёта мест: число прокатных комплектов в брони ограничено свободным прокатным фондом — `rental_count <= free_rental_kits`. Запись «своя экипировка» занимает место в группе, но не занимает прокатный комплект из фонда. | Must | R-003; Ответ № 2; глоссарий |
| **FR-15** | Система должна запрещать запись, превышающую лимит мест (FR-13) или лимит свободных прокатных комплектов экипировки (FR-14), исключая двойную бронь и овербукинг. | Must | Бриф (двойные записи); ответ № 2 |
| **FR-45** | Система должна использовать `price_total` из ответа API как итоговую стоимость брони (read-only; сервер — источник истины). Приложение не пересчитывает итог самостоятельно: при расхождении локальной оценки и серверного значения приоритет имеет `price_total` из ответа. | Must | R-005, R-010 |
| **FR-50** | Система должна позволять клиенту при записи указать текстовый комментарий к брони — аллергии и особенности питания. Комментарий передаётся в админку/шефу вместе с бронью. Поле опционально, максимум ~500 символов. | Must | Ответ № 17; бриф («мысли вслух» про аллергии) |

**Граница со серверной интеграцией.** Детали серверной реализации брони (транзакции, SLA, маппинг полей) — вне скоупа клиентского приложения. Существующий бэкенд выступает black-box источником истины; атомарность операций и гарантия «0 двойных броней» обеспечиваются на его стороне. Граница между приложением и бэкендом закрывается контрактом API; клиент полагается на коды ответов и значения полей из ответа (включая `max_seats`, `free_rental_kits`, `price_total`).

**Данные существующей инфраструктуры (R-015).** Проект учебный/тестовый, легаси-данных нет. Каноническая схема — контракт API (`01-analysis/api/`): бэкенд по условию предоставляет все поля контракта (`rental_price`, `studio_address`, `menu`, `free_rental_kits` и пр.). Требований к миграции, backfill и дозаполнению исторических слотов не формулируем — это вне скоупа клиентского приложения.

**Модель цены (учёт).** Итоговая стоимость брони рассчитывается сервером по модели: `price_total = price × seats_count + rental_price × rental_count`, где `price` — цена места на слоте, `seats_count` — число мест в брони, `rental_price` — цена проката одного комплекта экипировки, `rental_count` — число прокатных комплектов. Своя экипировка бесплатна; прокат доплачивается отдельным тарифом. Валюта — RUB. Цена фиксируется на момент создания брони и далее не меняется. Приложение отображает итог через `price_total` из ответа API (см. FR-45), не пересчитывая его локально.

**Работа со временем.** Время хранится и передаётся в API в UTC, а отображается клиенту в его локальной часовой зоне. Сервер — источник истины по времени и по статусу отмены (ранняя / поздняя / отменена студией). Граница ранней отмены — ровно за 24 часа до старта включительно (см. FR-17).

## Вне скоупа

Функции шефа, владельца/администратора, расписание-CRUD, онлайн-оплата и программа лояльности обеспечиваются существующей инфраструктурой или относятся к развитию — в клиентском приложении и его API не реализуются.

**Оценки шефов — Phase 2 (осознанное сужение скоупа, подтверждено заказчиком).** Оценки шефов не входят в MVP и переносятся в Phase 2. Это сознательное решение по сужению скоупа: заказчик ранее называл эту функцию критичной, поэтому она не отбрасывается, а планируется к реализации в Phase 2 — после запуска критичного набора MVP (просмотр слотов, запись, отмены) в жёсткий срок начала сезона. Перенос согласован с заказчиком. На MVP-скоуп и трассировку требований это не влияет.

**«Поделиться» классом — Won't (MVP) / Phase 2 (согласовано с заказчиком).** Функция «поделиться» слотом (share класса: deep-link/текст класса) не входит в MVP и переносится в Phase 2. На карточке класса (SCR-003) иконка «Поделиться» в MVP неактивна/декоративна; специфицировать поведение шеринга в MVP не требуется. Согласовано с заказчиком. На MVP-скоуп и трассировку требований это не влияет.
