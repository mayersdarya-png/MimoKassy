# Бизнес-требования

## Этап 2. Зачем бизнесу продукт: цели, метрики успеха, ограничения верхнего уровня.

Скоуп — клиентское мобильное приложение и API для него. Бизнес-цели, обслуживаемые существующей инфраструктурой (картина для владельца, интерфейс шефа, оценки), в этот список не входят.

**О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре). Нумерация намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных документах.

**Роли в текущей поставке.** В текущую поставку входит только роль **Клиент** (самостоятельная запись на кулинарные классы с мобильного приложения). Интерфейсы шефа и администратора/владельца — вне скоупа текущей поставки: они обеспечиваются существующей инфраструктурой. Это осознанное решение в рамках MVP-подхода и жёсткого срока: продукт фокусируется на клиентском сценарии, который снимает основную боль (хаос записи и двойные брони), а серверные/ролевые функции остаются на стороне существующего бэкенда.

| ID | Требование | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| **BR-1** | Устранить ручную путаницу и двойные брони, возникающие при ведении записи через WhatsApp и Google-таблицу. Учёт записей должен вестись в единой системе, исключающей конфликты по местам (столам) и по прокатной экипировке. | Must | Бриф (письмо Артёма, п. о хаосе в выходные и двойных записях); Описание домена → «Цель клиентского приложения» |
| **BR-2** | Дать клиентам возможность самостоятельно записываться на кулинарные классы онлайн без ручного посредничества владельца («люди сами записываются, а я только смотрю»). | Must | Бриф (письмо Артёма); Описание домена → «Цель клиентского приложения» |
| **BR-5** | Снизить простой столов и потери на закупленные продукты за счёт правила отмены 24 часа и фиксации поздних отмен (контроль без отталкивания клиентов — «ограничить, но не злить»). | Must | Бриф (проблема отмены за 10 минут до начала); Ответы заказчика № 7–8; Описание домена → «Политика отмен» |

| **BR-8** | На старте оплата остаётся офлайн (наличные / перевод); продукт лишь фиксирует запись и показывает цену. Онлайн-оплата — за рамками скоупа, заложить в развитие. | Must | Бриф (наличка/перевод, не уверены в онлайн-оплате); Ответы заказчика № 9, 11; Описание домена → «Ограничения» |
| **BR-9** | Клиентское приложение — мобильное (использование с телефона); взаимодействует с существующим бэкендом через API. | Must | Бриф (пользуются с телефона); Ответы заказчика № 19; Описание домена → «Технические/платформенные» |

## Метрики успеха

Целевые значения — разумные ориентиры для MVP (как в исходниках: точные пороги фиксируются как допущение, при наличии данных уточняются).

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :--- | :--- | :--- | :--- |
| **M-1** | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей за сезон). | ≥ 70% к концу первого сезона | BR-2 |
| **M-2** | Число двойных броней / конфликтов по местам (столам) и по прокатной экипировке. | 0 (полное устранение) | BR-1 |
| **M-4** | Простой столов из-за поздних отмен (доля поздних отмен от записанных). | Снижение относительно текущего уровня (целевой ориентир ≤ 10%) | BR-5 |
| **M-6** | Готовность к старту сезона. | Запуск в срок ~2 месяца с критичным набором функций (просмотр слотов, запись, отмены) | BR-7 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало следующего сезона (~2 месяца); бюджет ограничен → строго MVP-подход, «навороты» откладываются. (Бриф; Ответы заказчика № 20)
- **Оплата офлайн на старте.** Онлайн-оплата вне границ скоупа. (Ответы заказчика № 9, 11)
- **Mobile-first.** Клиентское приложение — с телефона; данные получает из существующего бэкенда через API. (Ответы заказчика № 19)
- **Вне скоупа.** Функции шефа/владельца, расписание-CRUD и админка обеспечиваются существующей инфраструктурой. Онлайн-оплата, программа лояльности, публичный рейтинг — направления развития. (Описание домена → «Границы скоупа»)
- **Оценки шефов — Phase 2 (подтверждено заказчиком).** Оценки шефов осознанно вынесены в Phase 2 и не входят в MVP. Заказчик ранее называл функцию критичной, поэтому она не отбрасывается, а планируется к реализации после запуска критичного набора MVP (BR-7) — это сознательное сужение скоупа ради попадания в жёсткий срок начала сезона. Перенос согласован с заказчиком. (Бриф; Описание домена → «Границы скоупа»; решение R-001)
