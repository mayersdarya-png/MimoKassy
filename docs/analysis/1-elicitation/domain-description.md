# Описание домена и ограничения

## Описание домена

### Бизнес

«Шеф-стол» — кулинарная студия в лофте на бывшем заводе, работает третий год. Студия проводит групповые кулинарные классы: гости под руководством шефа готовят ужин по заданному меню, а потом вместе его едят.

Класс длится около 3 часов и проходит по одной из программ: простая (например, базовая итальянская кухня — для новичков) или сложная (с духовкой, сувидом и прочей техникой — для опытных). Каждый класс ведёт **шеф** (сейчас их пятеро, в сезон подключаются приглашённые).

### Процесс

Владелец (Артём) заранее формирует расписание на неделю вперёд: для каждого класса задаются дата и время старта, программа (меню), назначенный шеф и число мест.

Студия располагает **12 рабочими столами**; для классов с духовкой и сложной техникой берут не более 8 человек (иначе шеф не успевает за всеми и не хватает плит), для простых классов — до 12 (по числу столов). Клиент видит свободные слоты и записывается, указывая, придёт ли он **со своим фартуком и набором ножей** или возьмёт **экипировку напрокат**. До класса клиент может отменить запись — при ранней отмене место освобождается, поздняя отмена (когда продукты уже закуплены) фиксируется без освобождения места.

Сейчас запись клиентов ведётся вручную (WhatsApp + Google-таблица), что приводит к двойным записям, путанице в выходные и ситуациям, когда гость приходит, а продукты на его порцию не закуплены.

### Цель клиентского приложения

Дать клиентам самостоятельную онлайн-запись на кулинарные классы (просмотр и фильтрация слотов, бронирование, выбор экипировки, отмена, свои бронирования, комментарий про аллергии), исключив ручную путаницу и двойные брони.

Управление расписанием и работу шефов/владельца обеспечивает существующая инфраструктура; приложение потребляет её данные через API.

### Ключевые объекты домена

- **Кулинарный класс / слот** — конкретное занятие группы: дата/время старта, программа (меню), шеф, всего и свободно мест. Создаётся в существующей инфраструктуре; клиенту доступен только для чтения.
- **Программа (меню)** — вариант класса (простая/сложная), определяет длительность, требуемое оборудование и потолок мест (справочные данные, read-only).
- **Шеф** — ведёт класс; для клиента — справочные данные слота (read-only).
- **Рабочий стол** — часть инвентаря студии (всего 12); определяет физический лимит группы.
- **Экипировка (фартук + набор ножей)** — прокатный фонд студии; может выдаваться клиенту напрокат.
- **Запись (бронь)** — резерв места клиентом на конкретный слот: число мест, своя/прокатная экипировка, комментарий (аллергии), статус.
- **Клиент** — записывается на классы, отменяет запись, указывает особенности питания.

## Пользователи и роли

Скоуп текущей работы — **только клиентское мобильное приложение и API для него**.

Роли Шеф и Владелец/Администратор обслуживаются существующей инфраструктурой (бэкенд, расписание, админка) и в этот скоуп не входят.

| Роль | Описание | Цели | В скоупе |
| :--- | :--- | :--- | :--- |
| **Клиент** | Гость студии, приходит готовить. Пользуется с телефона. | Увидеть и отфильтровать свободные слоты, записаться (себя и гостей), выбрать свою/прокатную экипировку, указать аллергии, видеть свои бронирования, отменить запись, получать напоминания. | **Да** |
| **Шеф** и **Владелец/Администратор** | Взаимодействуют с системой через уже реализованные интерфейсы существующей инфраструктуры. | — | **Нет** |

Клиентское приложение лишь потребляет подготовленные ими данные (слоты, программы, шефы) через API.

## Ограничения

### Бизнес-ограничения

- Всего **12 рабочих столов** — общий физический потолок групп.
- Простой класс (без сложной техники) — до 12 человек; класс с духовкой/сложной техникой — не более 8 человек.
- Лимит мест слота = `min(свободные столы, потолок программы)`.
- Лимит прокатной экипировки считается отдельно от лимита мест: клиент со своей экипировкой занимает место, но не расходует прокатный фонд.
- **Политика отмен**: бесплатная отмена не позднее чем за **24 часа** до старта (продукты закупаются заранее); поздняя отмена (< 24 ч) фиксируется без освобождения места и без денежных штрафов.
- **Оплата офлайн** (наличные / перевод на карту) — приложение лишь фиксирует запись и показывает цену.
- **Период списка слотов по умолчанию — 7 дней (R-027)**. По умолчанию клиенту показываются слоты на ближайшие 7 дней от текущего момента — это совпадает с горизонтом планирования расписания («на неделю вперёд»). Более длинный период доступен через фильтр дат. При отсутствии расписания на ближайшие дни показывается empty state («Пока нет доступных классов»).
- **Отмена класса студией / по форс-мажору (R-008)**. Если класс отменяет студия (организатор, срыв поставки продуктов, поломка оборудования), бронь на стороне клиента не удаляется, а получает статус **«Отменён студией»** с указанием причины; клиенту приходит push-уведомление. Повторная запись на отменённый слот запрещена. Само действие отмены слота выполняется в существующей инфраструктуре (вне скоупа клиентского приложения).
- **Аллергии и особенности питания** указываются клиентом в комментарии к записи (свободное текстовое поле) и передаются в админку/шефу. Сложной системы профилей аллергий в MVP нет.

### Технические / платформенные

- **Mobile-first**: клиентское приложение рассчитано на телефон.
- Клиентское приложение взаимодействует с существующим бэкендом через API; слоты, программы и шефы доступны только для чтения.
- **Граница серверной интеграции (R-004)**. Детали серверной реализации — транзакционность, SLA, маппинг сущностей на внутренние модели бэкенда — вне скоупа клиентского приложения. Существующий бэкенд — black-box источник истины: гарантия «0 двойных броней» обеспечивается на его стороне (атомарная проверка свободных мест и прокатной экипировки при бронировании). Со стороны клиента граница закрывается контрактом API: приложение полагается на ответы бэкенда и корректно обрабатывает отказ, если мест или прокатной экипировки уже нет.
- **Данные существующей инфраструктуры (R-015)**. Это учебный/тестовый проект, исторических данных и легаси-схемы нет. Канонической схемой считается контракт API (OpenAPI в `01-analysis/api/`): бэкенд по условию отдаёт ровно те поля, что описаны в контракте, включая `rental_price`, `menu`, `studio_address`, `free_rental_kits` и т. п. Поэтому миграция/backfill и стратегия дозаполнения исторических слотов вне скоупа — отдельных требований к ним не формулируем.

### Проект

- **Срок** — к началу следующего сезона, ~2 месяца (жёсткая цель).
- **Бюджет ограничен** → начинаем с самого нужного, «навороты» позже.

## Границы скоупа

Скоуп — клиентское мобильное приложение и API для него. Управление расписанием, роли шефа и владельца, оценки, админка — это существующая инфраструктура, вне скоупа.

| В скоупе клиентского приложения | Вне скоупа (существующая инфраструктура / не делаем сейчас) |
| :--- | :--- |
| Регистрация/вход клиента по имени и телефону | Роли Шефа и Владельца/Администратора, админ-экран |
| Просмотр и фильтрация списка слотов (дата, программа, наличие мест, шеф) | Создание и редактирование расписания (слоты приходят из существующего бэкенда через API) |
| Самостоятельная запись, в т. ч. несколько мест за раз (R-013) | Оценка шефа и рейтинги — Phase 2 (R-001; перенос согласован с заказчиком) |
| Выбор «своя / прокатная экипировка», контроль лимита мест и прокатного фонда | Отметка явки/неявки, отмена слота шефом + массовое уведомление |
| Комментарий к записи (аллергии / особенности питания) | Онлайн-оплата; ручная отметка «оплачено» |
| Список своих бронирований | Автоматический учёт аллергий по профилю |
| Отмена записи с правилом 24 часов | Программа лояльности для постоянных клиентов |
| Цена и фиксация записи (оплата офлайн) | Публичный показ рейтинга клиентам — Phase 2 (R-001) |
| Напоминания о записи — push за 24 ч и за 2 ч до старта (R-006) | SMS/email/inbox-напоминания — Phase 2 |

## Глоссарий

| Термин | Определение |
| :--- | :--- |
| **Кулинарный класс / слот** | Запланированное групповое занятие: дата/время старта, программа (меню), шеф, всего и свободно мест. |
| **Программа (меню)** | Вариант класса: простой (до 12 человек) или сложный, с духовкой/техникой (до 8 человек). |
| **Группа** | Состав клиентов, записанных на один слот. |
| **Рабочий стол** | Часть инвентаря студии; всего 12; определяет физический лимит группы. |
| **Экипировка** | Фартук + набор ножей из прокатного фонда студии. |
| **Аренда / прокат** | Выдача студийной экипировки клиенту, у которого нет своей. |
| **Своя экипировка** | Клиент приходит со своим фартуком и ножами; занимает место в группе, но не расходует прокатный фонд. |
| **Запись (бронь)** | Резерв одного или нескольких мест на конкретный слот: число мест, своя/прокатная экипировка, комментарий (аллергии), статус. |
| **Аллергии / особенности питания** | Текстовый комментарий к записи, передаваемый шефу и админу. |
| **Отмена** | Снятие записи. Клиентом: ранняя (≥ 24 ч до старта) освобождает место; поздняя (< 24 ч) фиксируется без освобождения места. Студией (организатором / по форс-мажору): бронь переходит в статус «Отменён студией» с причиной, клиент получает push, повторная запись на этот слот запрещена. |
| **Отмена студией** | Отмена класса организатором или из-за форс-мажора (срыв поставки, поломка оборудования) — выполняется в существующей инфраструктуре. На стороне клиента бронь сохраняется со статусом «Отменён студией» + причина; отправляется push-уведомление; повторная запись на отменённый слот недоступна. |
| **Администратор / владелец** | Артём — управляет расписанием и шефами через существующую инфраструктуру (вне скоупа клиентского приложения). |
| **MVP** | Минимально жизнеспособный продукт: набор функций к старту сезона. |
