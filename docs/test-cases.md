# Тест-кейсы для приложения «Мимо Кассы»

**Проект:** Клиент-серверное приложение для записи на кулинарные классы  
**Дата:** 2026-07-06  
**Всего тест-кейсов:** 15

---

##  Промпт для генерации тест-кейсов

Ты — senior QA engineer с 10-летним опытом тестирования мобильных и веб-приложений. 
Твоя задача — написать подробные тест-кейсы для приложения «Мимо Кассы» — сервиса 
для записи на кулинарные классы.

Опирайся на следующие требования и документацию:

1. Приложение состоит из:
   - Мобильного клиента (Android, Kotlin + Jetpack Compose)
   - Бэкенда (Python + FastAPI + PostgreSQL)
   - Авторизация по номеру телефона с подтверждением по SMS (в тестовой среде код 123456)

2. Основные пользовательские сценарии (use cases):
   - UC-1: Запись на класс (выбор слота, количества мест, экипировки, подтверждение)
   - UC-2: Отмена записи (ранняя и поздняя, с правилом 24 часов)
   - UC-3: Просмотр и фильтрация списка классов

3. Ключевые фичи, которые нужно покрыть тестами:
   - Регистрация и вход
   - Просмотр списка классов
   - Карточка класса
   - Запись на класс (включая ошибки: нехватка мест, нехватка экипировки)
   - Отмена записи (ранняя / поздняя)
   - Мои бронирования (список, детали)
   - Профиль (редактирование, выход)

4. Формат тест-кейса:
   - ID (TC-XXX)
   - Название
   - Приоритет (Critical / High / Medium / Low)
   - Предусловия
   - Шаги (нумерованный список)
   - Ожидаемый результат
   - Фактический результат (заполняется после выполнения)

5. Важные бизнес-правила, которые должны быть проверены:
   - Лимит мест в группе: простая программа — 12 человек, сложная — 8 человек
   - Прокатная экипировка: всего 12 комплектов
   - Отмена: ранняя ≥ 24 часа до старта (места возвращаются), поздняя < 24 часа (места не возвращаются, штрафов нет)
   - Максимум мест в одной брони: 3 (себя + до 2 гостей)
   - Своя экипировка — бесплатно, прокат — отдельный тариф

6. Критерии качества тест-кейсов:
   - Каждый тест-кейс должен быть атомарным (проверяет одну конкретную функцию)
   - Шаги должны быть чёткими и воспроизводимыми
   - Ожидаемые результаты должны быть измеримыми
   - Покрыть как позитивные, так и негативные сценарии
   - Включить граничные случаи (edge cases)

7. Добавь статистику по тест-кейсам и примечания.
