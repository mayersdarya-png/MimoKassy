## 1. Промпт №1: Полный — генерация тест-кейсов


Ты — senior QA engineer с 10-летним опытом тестирования мобильных и веб-приложений. 
Специализируешься на тестировании e-commerce и сервисных платформ.
Твоя задача — написать подробный набор тест-кейсов для приложения «Мимо Кассы».
«Мимо Кассы» — сервис для записи на кулинарные классы. Состоит из:
- Мобильного клиента (Android, Kotlin + Jetpack Compose)
- Бэкенда (Python + FastAPI + PostgreSQL)
- Авторизация по номеру телефона с OTP-кодом (в тестовой среде код: 123456)
Ожидаемый результат набор из 15–20 тест-кейсов, сгруппированных по модулям:
- Авторизация
- Классы (просмотр)
- Запись
- Бронирования (отмена)
- Профиль
- Фильтрация

Результат должен быть готов к использованию в QA-процессе.

## 2. Промпт №2: Тест-кейсы для одного модуля (бронирования)

На основе предыдущего набора тест-кейсов, создай отдельный набор тест-кейсов 
только для модуля «Бронирования».

Сценарии, которые нужно покрыть:

1. Просмотр списка броней (предстоящие / прошедшие)
2. Просмотр деталей брони
3. Отмена брони (ранняя)
4. Отмена брони (поздняя)
5. Повторная отмена брони (недоступна)
6. Отмена брони после старта класса (недоступна)

Важные проверки:
- Правильное разделение на «Предстоящие» и «Прошедшие»
- Правильный статус после отмены (cancelled / late_cancel)
- Возврат мест при ранней отмене
- Невозврат мест при поздней отмене
- Нет штрафов при поздней отмене
- Кнопка «Отменить» недоступна, если бронь уже отменена или класс начался

Формат — такой же, как в предыдущем промпте. Ожидаемое количество — 6–8 тест-кейсов.

## 3. Промпт №3: Негативные тест-кейсы


На основе предыдущего набора тест-кейсов, создай отдельный набор только для 
негативных тестовых сценариев.

Негативные сценарии:

1. Вход с неверным кодом → сообщение об ошибке, доступ к приложению закрыт
2. Запись на класс — нехватка мест → сообщение с указанием свободных мест
3. Запись на класс — нехватка прокатной экипировки → сообщение с указанием доступных комплектов
4. Запись на класс, который уже начался → ошибка, запись недоступна
5. Запись на отменённый студией слот → ошибка, запись недоступна
6. Повторная отмена брони → кнопка неактивна, сообщение «Запись уже отменена»
7. Отмена брони после старта класса → кнопка неактивна, отмена недоступна
8. Попытка записать 4 места (больше максимума) → ограничение на UI / ошибка валидации
9. Попытка записаться на 0 мест → кнопка неактивна
10. Попытка входа с пустым номером телефона → кнопка неактивна / валидация

Для каждого сценария опиши предусловия, шаги и ожидаемый результат (сообщение об ошибке, состояние UI).

Формат — такой же. Ожидаемое количество — 10 тест-кейсов.




## 4. Промпт №4: Генерация тест-кейсов из требований


У меня есть функциональные требования (FR) для приложения «Мимо Кассы». 
На основе FR-XX создай тест-кейсы.

Вот требования:

FR-10: Система должна позволять клиенту записаться на выбранный слот (себя и гостей). 
После подтверждения запись появляется в «Моих бронированиях».

FR-12: Клиент может записать себя + до 2 гостей (максимум 3 места).

FR-13: Лимит мест в слоте ограничен min(free_seats, program.capacity_cap, 3).

FR-17: Ранняя отмена (≥ 24 ч до старта) — места возвращаются в слот.

FR-18: Поздняя отмена (< 24 ч до старта) — места НЕ возвращаются, штрафов нет.

Для каждого из этих требований создай по 1–2 тест-кейса:
- Позитивный (успешное выполнение)
- Негативный (попытка нарушить правило)

Формат — такой же, как в основном промпте.

## 5. Промпт №5: Проверка граничных случаев


Создай набор тест-кейсов для проверки граничных случаев (edge cases) 
в приложении «Мимо Кассы».

Граничные случаи:

1. Время до старта ровно 24 часа — должно считаться ранней отменой (места возвращаются)
2. Время до старта 23 часа 59 минут — должно считаться поздней отменой (места НЕ возвращаются)
3. Свободно мест ровно 3 — можно записать 3 места
4. Свободно мест ровно 2 — можно записать максимум 2 места
5. Свободно мест 0 — кнопка «Записаться» неактивна, пометка «Мест нет»
6. Свободно прокатных комплектов 0 — опция «Прокатная» недоступна
7. Запись 1 место (сам клиент) — минимально допустимое
8. Пустой список классов — отображается empty state с подсказкой
9. Пустой список броней — отображается empty state «У вас пока нет записей»
10. Фильтры — ни одного класса под условия → empty state с предложением сбросить фильтры

Для каждого случая опиши:
- Предусловия (как добиться такого состояния)
- Шаги
- Ожидаемый результат (поведение UI и бизнес-логики)

Формат — такой же. Ожидаемое количество — 8–10 тест-кейсов.

## 6. Промпт №6: Тест-кейсы для безопасности и авторизации

### Следующий промпт (фокус на безопасности)

Создай набор тест-кейсов для проверки безопасности и авторизации 
в приложении «Мимо Кассы».

Сценарии:

1. Доступ к защищённым эндпоинтам без токена — 401 Unauthorized
2. Доступ с истёкшим access-токеном — 401, автоматическое обновление через refresh
3. Доступ с невалидным токеном — 401
4. Выход из аккаунта — токен инвалидируется, доступ к защищённым данным закрыт
5. Попытка доступа к чужой брони — 403 Forbidden (или 404, если скрываем существование)
6. Попытка удалить чужой аккаунт — 403 Forbidden
7. Rate limit на OTP (не более 1 запроса в 30 секунд) — 429 Too Many Requests
8. Лимит попыток ввода кода (≤ 5) — блокировка, требуется повторная отправка

Для каждого сценария опиши:
- Предусловия
- Шаги (включая HTTP-запросы, если нужно)
- Ожидаемый результат (код ответа, сообщение, поведение UI)

Добавь проверку, что пользователь видит только свои данные (NFR-12).

Формат — такой же. Ожидаемое количество — 6–8 тест-кейсов.
